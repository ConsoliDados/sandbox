# ADR-0007 — State storage follows the XDG Base Directory spec

- **Status:** Accepted
- **Date:** 2026-05-06
- **Phase:** 1

## Context

`sandbox` has three kinds of persistent state:

1. **Config** — user-edited, backed up, version-controlled in dotfiles (e.g. `config.toml` with profile definitions, `languages/*.toml` overrides).
2. **State** — per-project metadata (container hash, named volumes, ports, last run timestamp). Semi-permanent; survives across runs but can be regenerated by re-running.
3. **Cache** — derived data (scan results keyed by source hash). Disposable; rebuilt on demand.

These have different durability and backup expectations. Mixing them in one directory is messy.

## Decision

We follow the XDG Base Directory Specification. Paths are resolved via the `directories` crate (handles platform defaults and env var overrides).

| Kind | Variable | Default | Used for |
|---|---|---|---|
| Config | `$XDG_CONFIG_HOME` | `~/.config/sandbox/` | `config.toml`, user `languages/`, dotfiles |
| Data | `$XDG_DATA_HOME` | `~/.local/share/sandbox/` | `containers/<hash>/`, `proxy/` (Traefik state) |
| Cache | `$XDG_CACHE_HOME` | `~/.cache/sandbox/` | `scan/<hash>.toml`, derived hashes |

## Alternatives considered

- **(a) Single directory** under `~/.config/sandbox/` (everything together). Rejected: mixes user-edited config with disposable cache; backup tools that target `~/.config/` would copy gigabytes of scan cache; Linux convention violated.
- **(b) Custom location via `SANDBOX_HOME` env var.** Rejected as the only mechanism: not idiomatic on Linux, surprising to users who expect XDG compliance. We do honor `SANDBOX_HOME` as an override that points to a single root, but only as a developer/CI convenience.

## Consequences

Positive:
- Backup tools and dotfile managers can target the config dir without dragging cache or stale state.
- Cache is safely deletable (`rm -rf ~/.cache/sandbox/`) without losing config.
- macOS port (future) gets sensible defaults from the `directories` crate (`~/Library/Application Support/sandbox/`, etc.).

Negative:
- Three directories to keep a mental model of. Mitigated: `sandbox config path` prints all three; the SRS documents the layout.
- Users who symlink `~/.config/sandbox/` to a dotfile repo cannot symlink the cache without confusion. This is the right outcome — caches don't belong in version control.
